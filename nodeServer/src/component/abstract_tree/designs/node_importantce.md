# 节点的定义

节点是一种妥协，其是向量数据库失败的体现

如果能照抄人的这部分，那么词汇与词汇之间是平等的而无父子关系，父子关系是后加的，运行时的。词与词，或者说对象与对象，概念与概念的串联是靠概念本身，即概念本身是概念和概念的连接。之所以将这个复杂系统化为抽象树的方式实现，是因为概念与概念之间的连接，那个想象界构建的过程，已经是我能力的边界，我不知道具体的检索机制，不知道是哪种算法使得想象界成立，所以我只能偶性的设立一种方法，试图去复刻那一结果

概念与概念之间的关系（这只是为了方便管理）有：

```list
关联关系
抽象化与具象化的关系（父子关系）
同位关系（在同一个父亲下的关系）
两面关系（同一个东西在两个理论下的展现和分析结果）
指涉关系
```

但是其实一切关系都是一种指涉关系，概念依靠指涉其他概念而成立，这一指涉指的是通过想象其他概念的在场，使这个场成为目的概念在场就能完成的场，当然目的概念不一定是核心目的，更多的是诸多理论体系尝试构造

这一构造的习惯即这一构造能依赖的唯一方式，正是这一唯一方式决定了体系化的固定，因为其是唯一的，而且其常常也在自指中被肯定其唯一性

还有就是，如果一个理论体系是唯一的，在大框架确认下来以后，其细枝末节也必定是可以被唯一还原的

受限于LLM现有的，只能整段输出文本的，基于一定预设与常识，试图通过常识人的视角接受如此预设的，特性

概念与概念间的索引，如果将其分为LLM（main_virtual）内和LLM外的维护系统，则LLM内必须接受的只能是“预设”，这一预设必然是叙述句，其必然是某些事实的呈现，这才能成为预设，如果是疑问句则会成为欲望，但是叙述句也组成了欲望

而预设最适合规范管理的方式就是抽象对象树，这样能比较好的管理对象，而且天生拥有逻辑推理能力，因为概念需要依靠其他概念而存在，那么对于包含两个概念的关系，亦或者包含多个概念的理论，每个概念里都包含了完整的关系或者理论，其实就有重复引用之嫌

解决的方法是把一个节点分为标志和内容，标志决定了节点如何成为独一无二的，而不用还原出其内容。而内容一般要依赖其他节点存在，塞入上下文的时候可以用a:theory1,b:theory1;theory1:xxx来表述
而理论则会完整的描述其他对象，通过这个可以引入其他对象

**需要注意的是，某些概念，或者说大多数概念其实是在常识中的，一旦这个概念属于在常识中的概念，那么其包裹在theory中的时候就不用附加，而判定是否属于常识，又靠检查抽象树里存在这玩意吗。如果一个概念在theory里被冠以了常识外的定义，那么这个概念就需要加入抽象树**

**即使是常识，在设计时也需要和其他概念一视同仁，一旦改写了该概念，那么该概念也就需要加入抽象树，据此，我们从动力学上不难得出抽象树其实就是被theory改写的常识**

由上，不难发现子节点和父节点的关系是子节点必然适应父节点的theory，而且子节点必须有自己的theory
对于需要引入的概念，如果其可以视为抽象树的某节点的子节点，而且其并没有新的theory,则其不写入抽象树，但是在模型的上下文里面其仍然作为该父节点的子节点，又由于抽象树的特性，父节点的孙节点，曾孙节点等其实也可以作为父节点的直接子节点，那么树结构只是为了theory的压缩，本质上，可以认为整个树不存在所谓结构

接下来需要考虑theory的问题，theory当然可以作为节点，但是theory和节点还是有本质区别的，当theory作为节点，其必须作为其他theory想象中的成员，但是当theory发挥效用的时候，其自身就是自身完美的想象界
于是可以通过节点管理被指代的theory,theory作为节点首先被自己管理，因为其已经作为节点，于是其一定被另一theory描述或者其父节点被theory描述。当然，如果一个theory仅仅被当theory使用，那么其不用管这个theory作为节点的被theory，只有在theory被另一theory主动网罗的时候，其才需要作为节点呈现，theory作为节点的时候，其以及其父可能会携带包括自身的多个theory，这玩意很占上下文空间，所以没必要引入

以此类推，如果是非theory节点，引入的时候其父的theory也没必要展开，其自己的theory也没必要展开，theory的定义本身就决定了在这个意义下的全部额外信息，额外解释，非常识的部分，只用引入主要的theory就行

当然，theory的片面性对设计者或者使用者来说造成了一种欲望，那么就需要将此欲望转换为agent层面上的驱力。作为人类，我们希望对体本身以完整的theory结构为前提，那么这就意味着theory必须全量载入上下文，且事实更新。但是实际上，theory是依赖有限的上下文，且通过有限次的，有限定条件的，某些时刻进行的更新与生效。于是完整的theory结构就必须等同于一套依赖的算法，这套算法需要将任何问题映射为一或多个通过有限的theory能解决整体问题的的对话窗口。

问题为什么需要theory才得以解决？theory是常识外覆盖的基础，theory提供了对事物关系的理解，theory必然创造新概念或者说常识概念的新定义，否则其就不应该被添加为theory，但是theory里也可以只有一个概念，因为LLM的常识并不是一个统一的theory，完全可以通过一个描述（theory）来固定常识，使常识超出常识，从而创造新的概念。虽然在此过程中，除了被创造的概念以外，其他常识也离开了”LLM的常识“，但是其的离开本就蕴含在”LLM的常识“中，所以不用添加为概念，只有通过"LLM的常识"来固定自身，超脱自身，以“新视角”来看自身，才成为了需要添加的概念。
如果在一个讨论中，或者说一个思考中，运用到了某些概念，那么只需要这个概念最近的上的theory，至于theory的theory，那是对theory的反思时需要用到的。于是，假设思考中存在对象object[],对于object[i]存在object[i].parent_theories[],而parent_theories里，正常情况下只需要出现一个
这是因为，多theory分析实质上是通过多theory结合常识构造一个大的theory，常识一面维持着这多个theory，作为通用的残差让每个theory成立的同时，也让多个theory联合起来；另一面，常识也分割开了期望的大theory体系，使得theory方便管理和计算
那么object[i].parent_theories[]中出现的那一个，就作为这一综合theory网络中的一个节点，其的存在目的，或者说被引入的目的，即维护这个总的theory，所以多个之间互不干涉。

就上文所言，这一综合theory网络其实不可拆分，其倒是可以通过分发子任务和删除已经得出结论的theory来实现压缩上下文，但是那是后期的事情了，我们先不讨论，而且这玩意也好做

于是，将刚刚的需求映射到工程层面，我们一定就有：

```list
1.在对话时有的已经引入好的某些对象和其theory
2.在对话前就有的纷繁的抽象树和theory库
3.在对话时不停维护纷繁的背景的抽象树和theory库
4.在对话时不停维护的当下对话的抽象树和theory库
5.在对话结束时或者时刻消逝意义上维护的当前对话的抽象树和theory库及其持久化转移
6.使抽象树，theory库等数据结构有其意义，并且能一直按其意义行走的“算子”
7.应对抽象树，theory库纷繁性为有限性的机制
8.抽象树和theory库的原生默认结构
9.自我认知的建构
10.能建立当下境况，以第三人称审视共情等等功能的机制（即为主体性实践的结果兜底）
```
